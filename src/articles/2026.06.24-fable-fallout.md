## The Big Picture

The June 24 folder points to the same shift from several angles. The Fable 5 screenshots emphasize a jump in agentic coding performance. The token-maxxing screenshots emphasize that agentic workflows can multiply total tokens per task even when token prices fall. The workflow-pattern diagram shows the new operating shape: not one chat window, but classifier agents, fan-out agents, verifiers, tournaments, filters, and loops.

![The cost of intelligence is collapsing](/weekly-screenshots/2026.06.24/intelligence-cost-collapse.jpg)

That missing baseline matters: the supplied Stanford AI Index chart shows GPT-3.5-level inference falling from $20 per million tokens in November 2022 to $0.07 by October 2024, a 280x drop in roughly two years. The tension is that cheaper intelligence does not automatically mean cheaper tasks when agents multiply the number of tokens, tools, retries, and checks behind one finished result.

The reported SpaceX-Cursor deal adds another angle: coding tools are no longer just apps. They are becoming infrastructure bets, where the editor, model, GPU supply, and workflow harness all start to collapse into one strategic surface.

In other words, the new AI question is not "which chatbot is best?" It is "which workflow gets the result, proves it, and stays affordable?"

That is a very different game for builders. A single expensive model can be worth it for a hard task. It can also become a fragile dependency. A cheap model can save money. It can also waste money if it triggers more retries, more context stuffing, and more human cleanup.

The frontier is becoming a routing problem.

## Past two weeks

On June 17, reporting around the Fable 5 and Mythos 5 restriction described a Commerce Department licensing requirement for access by foreign persons, with national-security language around military-intelligence use. That turned model access from a product setting into a policy surface.

On June 17-22, token maxxing became a more concrete cost story. Economic Times framed token maxxing as the habit of treating token consumption as a proxy for value, while warning that unchecked token use can become the next cloud-spend problem. The folder's token diagrams make the same point visually: agents, tools, retrieval, validation, and retries can turn one simple request into a much larger token loop.

On June 17-22, GLM-5.2 entered the conversation as a recent open-model pressure point. Reporting described it as a Z.ai model aimed at long-horizon coding and agentic engineering work. That matters because open or locally runnable models become more attractive when closed frontier access feels politically or commercially unstable.

On June 17-19, reporting said SpaceX was buying Anysphere, the company behind Cursor, in a $60 billion all-stock deal set to close later pending regulatory approval. The strategic logic was compute: Cursor had the product and developer workflow, while SpaceX had the GPU infrastructure. That makes AI coding tools feel less like standalone software and more like vertically integrated compute businesses.

And the Claude Code workflow article linked for this post gives the practical frame: for complex tasks, the harness matters as much as the model. Claude Code dynamic workflows are designed to create task-specific multi-agent harnesses instead of forcing planning, execution, and verification into one long context window.

## Access Shock

The Fable 5 screenshot in the folder frames Fable as a major agentic-coding jump: 80.3% on SWE-Bench Pro and 29.3 on FrontierCode in the chart shown, ahead of the other models pictured.

That makes the reported access disruption more important. If the strongest model in a workflow becomes restricted, API-only, region-limited, or temporarily unavailable, the workflow itself becomes the risk surface.

The immediate lesson for local builders is not to panic about one model. It is to stop building around one irreplaceable model. Use the best frontier system when it earns the cost, but keep the workflow portable enough to route around access changes.

The second lesson is trust. A high-performing model still needs a safety story, an access story, and a verification story. The Fable fallout is a reminder that "best model" and "deployable model" are no longer the same question.

## New Realities

### Coding Tools As Infrastructure

The SpaceX-Cursor reporting belongs next to Fable and token maxxing because it points at the same new reality. Coding agents are not just interface products anymore. They are becoming compute, model, and workflow businesses.

If a coding tool controls the editor, the model lane, the context layer, the verification loop, and the GPU supply behind it, the competitive advantage shifts from app polish to infrastructure control.

### Workflow Harnesses

The Claude Code article, "A harness for every task," argues that Claude Code can now generate dynamic workflows: small JavaScript harnesses that spawn and coordinate subagents for the task at hand. The key idea is that a complex task should not always be solved inside one long-running chat context.

![Six workflow patterns](/weekly-screenshots/2026.06.24/six-workflow-patterns.png)

The six highlighted workflow patterns are:

- Classify-and-act
- Fan-out-and-synthesize
- Adversarial verification
- Generate-and-filter
- Tournament
- Loop until done

The important highlight for this post is **deep verification**. Instead of asking one model to write a report and then vaguely "check it," a workflow can identify factual claims, spawn separate agents to verify each claim, and even send those source checks through another verifier. That directly answers two failure modes from the last two weeks: too much trust in one model, and too much spending on unverified loops.

![Deep verification workflow](/weekly-screenshots/2026.06.24/deep-verification-workflow.png)

The article also warns that dynamic workflows can use more tokens and are best for complex, high-value tasks. That belongs right next to the token-maxxing discussion: workflows are powerful, but the harness has to include a budget and a stop condition.

### Verification As Product Work

The folder's "Goal Achieved" screenshot is a good local example of what verification should look like: named checks, final reruns, diff hygiene, skipped-lane caveats, token usage, and elapsed time.

That is the shape to copy. Do not just ask whether the AI finished. Ask what it ran, what passed, what was skipped, what changed, and how much the loop cost.

## Action Steps

### Token Maxxing

Stop using token volume as a productivity score. The better metric is outcome per token: bugs fixed, claims verified, tickets closed, documents shipped, customers helped, or dollars saved.

Put budgets on agentic tasks. A workflow that fans out five agents, then adds retrieval, validation, retries, and another judging pass should have a planned token range before it starts.

Separate learning spend from production spend. It is fine to explore heavily while finding the workflow. Once the workflow repeats, compress the prompt, cache the context, save the harness, and cut the waste.

Use deep verification selectively. It is worth spending more tokens to verify high-stakes claims, security findings, migrations, contract changes, and public-facing analysis. It is probably not worth running a panel of agents for a tiny copy edit.

Track the full cost of the loop, not just model price. The folder diagram says it cleanly: token price can fall while total task cost rises if tokens per task explode.

### GLM local / cloud inference

Treat GLM-5.2 as a current routing option to evaluate, not an automatic replacement for frontier APIs.

Start in cloud inference when you want quick comparison: latency, coding quality, context behavior, and cost per task. Move toward local or self-managed inference when the workload is repetitive, sensitive, high-volume, or needs resilience against provider access changes.

Benchmark it on your own work. Use the same prompt set across your preferred frontier model, GLM-5.2, and a cheaper fallback. Compare not just answer quality, but retries, verification failures, latency, and total tokens.

Keep a model router. Local GLM, hosted GLM, Claude, OpenAI, Gemini, and cheaper small models should sit behind the workflow, not be hard-coded into it. The workflow should decide which model gets which job.

Pair GLM testing with the Claude workflow patterns. For example: classify with a cheap model, fan out research with GLM, synthesize with a frontier model, and send factual claims through adversarial verification before publishing.

## Rumors

Fable access may come back with tighter controls, API-only lanes, regional limits, or additional safety gates. The rumor to watch is not the exact mechanism; it is the direction of travel.

Subscription generosity may tighten. The folder's subscription-vs-API screenshot argues that heavy users can consume far more than their monthly plan price in implied API value. If that math gets too lopsided, expect plan changes, caps, model gating, or priority rules.

Open coding models may become the default hedge. GLM-5.2's timing matters because it arrived during the same window as Fable uncertainty and token-cost pressure.

GPT-5.6 chatter is now pointed at Thursday, June 25, 2026. Treat that as unconfirmed until OpenAI posts it directly, but plan for the possibility that a new OpenAI release could reset the routing map almost overnight.

Workflow quality may become more important than model identity. The six-pattern diagram is the most durable idea in the folder: classify, fan out, verify, filter, compete, and loop until the stop condition is real.

The practical takeaway: use the best model you can justify, route everything else, and verify the output before it leaves the building.
